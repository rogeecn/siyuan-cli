# 把一个能跑的 CLI，做成一个真正可用的工具

有时候一个工具开始的时候，看起来真的很小。

我一开始做这个 SiYuan CLI 的时候，脑子里的想法其实很简单：把常用 API 包一层，做成命令行，能查、能写、能导出，平时自己用起来顺手一点。它不像做一个完整产品那样有庞大的界面、复杂的交互和漫长的需求讨论，它甚至给人一种错觉——好像只要把参数收好，把请求发出去，把返回结果打印出来，事情就差不多结束了。

后来我发现，恰恰是这种“看起来不大”的工具，最容易让人掉进一种很危险的自信里：我以为它已经可以用了。

## 真正的分界线，不是“能跑”，而是“敢不敢在真实环境里跑”

做这个工具的过程中，我最深的体会是，mock 世界和真实世界之间，隔着的不是一点点误差，而是一整层工程现实。

在本地测试里，很多东西都显得非常合理。接口返回 JSON，对象结构清清楚楚，字段名也都像文档里写的那样整齐。测试一绿，构建一过，命令一执行，心里很容易生出一种轻微但危险的满足感：差不多了。

可一旦把它真正接到 SiYuan 的真实实例上，那种“差不多”立刻就会原形毕露。

有的接口成功了，但响应体是空的；有的接口返回的是字符串，不是对象；有的接口根本不是我以为的那条；还有的接口文档写得没错，但当前实例里的实际行为又是另外一种样子。最开始碰到这些问题的时候，我其实是有点挫败的。因为这类问题不会给你一种特别戏剧性的“程序崩了”的感觉，它更像是在反复提醒你：你理解的系统，还不够真实。

而工具这种东西，最怕的其实不是报错，最怕的是一种“看上去像能用，但在关键一步不可靠”的状态。

## 我慢慢明白，工具感本质上是一种信任感

这次把 CLI 一路做下来的过程中，我越来越强烈地感觉到，所谓工具感，和功能多少关系没那么大，和信任感关系更大。

一个命令行工具，真正让人愿意一直用，不是因为它参数多，也不是因为它输出漂亮，而是因为你知道：当你敲下这条命令，它大概率会按你以为的方式工作。如果它失败了，失败信息也是真实的、可追踪的，而不是一种模棱两可的噪声。

所以后面我做的很多工作，看上去都不像“开发新功能”，更像是在给这个工具一点一点地补信任。

比如处理空响应。

在纸面上看，这只是一个很细的兼容问题：后端 `200 OK`，但是 body 为空。如果只是把它当成某个角落里的小 bug，那修法会很随意。但如果把它放在“信任”这个角度看，你就会发现这其实非常关键。因为对用户来说，空响应不应该被解释成一串莫名其妙的 JSON 解析错误。工具应该知道，这类情况在真实世界里是会发生的，并且应该给出稳定、可预期的行为。

再比如 `doc create` 返回字符串 ID 而不是对象、`doc update` 实际上应该走块更新接口、`file write` 需要 multipart、`file tree` 实际上应该走 `readDir`。这些事情单看都很碎，很像某种“接口适配杂活”。但我做着做着开始意识到，真正的工程工作本来就是这样：很多价值，不体现在你额外做了多少炫目的东西，而体现在你有没有把这些看起来不起眼、但会直接破坏使用体验的裂缝一条条补上。

## 我最深的变化，是不再轻易相信“差不多”

如果说做这个工具之前，我对工程的直觉更偏向“先做出来再说”，那做完这一轮之后，我对“验证”这件事的感受变得非常强。

我现在会更自然地把“它在测试里过了”跟“它在生产里可用”看成两件完全不同的事。

前者只能说明我写出来的假设体系内部是自洽的，后者才说明这个工具真的接上了现实。

这个变化其实不只影响了我怎么看待 CLI，也影响了我怎么看待所有工具型工作。以前我更容易把测试通过、接口打通、输出正常，当成阶段性终点。现在我会下意识再追问一句：那真实环境里呢？真实用户最容易踩到的路径呢？文档和当前行为不一致的时候，工具站在哪一边？

这种追问有时候会让事情变慢，因为它逼着你承认很多东西还没有真的结束。但它也会让结果变硬。那种“我知道这个命令不只是能跑，而是已经在真实环境里被我一轮一轮验过”的踏实感，和只看测试绿条完全不是一种东西。

## 做到后面，写代码反而不是最难的部分

很有意思的是，这一轮做到后面，我反而越来越觉得最难的并不是代码本身。

真正难的是判断。

判断这是实现 bug，还是我用错了命令；判断这是文档变了，还是当前实例的返回契约在漂移；判断这里应该在 HTTP 层兜底，还是在服务层归一；判断一个问题是“功能不优雅”，还是“生产环境不能使用”；判断什么时候应该继续扩测，什么时候应该停下来先修正在眼前已经被证实的问题。

这种判断不是靠多写几行代码就能得到的，它来自一轮又一轮地读错误、抓原始响应、回头对文档、再去真实环境里重跑。说得更直白一点，很多时候不是代码把问题修好了，而是你终于把问题看清楚了。

我想这也是这次做这个工具给我最大的提醒：工程质量不只是实现能力，也是一种识别现实的能力。

## 我最后得到的，不只是一个 CLI

如果只看结果，这轮工作当然是把一个 SiYuan CLI 打磨得更能在真实环境里用了。很多之前跑不通的命令现在都能真正跑起来，很多原本只在测试里成立的假设，也被改造成了真实可用的兼容逻辑。

但如果只把这件事理解为“我做完了一个工具”，我觉得又有点低估了它。

更准确一点说，这次过程让我重新理解了“可用”这两个字。

可用不是功能表上的一个勾，也不是 README 里的一个例子，更不是测试全部通过时屏幕上那一片绿色。可用是一种更朴素也更严格的东西：你把它接上真实系统，带着真实数据、真实返回、真实偏差和真实意外去跑，它依然不会轻易背叛你。

而一个工具一旦到了这个阶段，它才开始真正像一个工具。

我想，做完这个 SiYuan CLI 之后，我最大的感受并不是“又完成了一个项目”，而是我对工程里的很多小事变得更认真了。那些空响应、那些字段差异、那些文档和现实之间的小缝，看上去都不体面，也不浪漫，但恰恰是这些地方，决定了一个工具最后到底只是“看起来能用”，还是“真的能用”。

如果说这次做这个工具给我留下了一个最具体的收获，那大概就是：以后再遇到任何一个看起来已经差不多的系统，我都会提醒自己，多问一句——它真的在现实里站住了吗？
